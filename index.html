Before You Begin

What is This Lab About?

Encryption is the process of scrambling data so only authorised parties can read it. This lab explores symmetric encryption — where the same key encrypts and decrypts. You will move from the ancient Caesar cipher all the way to discovering real vulnerabilities in modern AES.

📄
Plaintext
The original readable data — your message, file, or image before encryption.
🔒
Ciphertext
The scrambled output after encryption — looks like random bytes to anyone without the key.
🗝️
Key
The secret controlling encryption and decryption. Everything depends on keeping this secret.
🎲
IV (Init Vector)
A random value mixed into the first block so the same plaintext encrypts differently each time.
Lab ToolAll modern cipher tasks use openssl enc on the Linux terminal. The Caesar tasks use Python 3. All files live in ~/Labsetup/Files/.

Task 1 — Classical Cryptography

Caesar Cipher — Encryption & Decryption

The Caesar cipher shifts every letter by a fixed amount (the key). With key = 3: A→D, B→E, …, Z→C. Decryption shifts backwards by the same amount. The skeleton file 1_caesar_skeleton.py provides the alphabet utility functions — you only fill in the logic.

Caesar shift — Key = 3

Complete Solution — caesar_encrypt and caesar_decrypt

1_caesar_skeleton.py — completed functions
def caesar_encrypt(plaintext, key):
    """Encrypt plaintext using Caesar cipher with given key."""
    result = []
    for char in plaintext:          # iterate over every character
        if char == ' ':               # spaces are kept unchanged
            result.append(' ')
        else:
            n = char_to_num(char)    # 'A'→0, 'B'→1, … 'Z'→25
            n = (n + key) % 26      # shift forward, wrap at Z
            result.append(num_to_char(n))
    return ''.join(result)


def caesar_decrypt(ciphertext, key):
    """Decrypt ciphertext using Caesar cipher with given key."""
    result = []
    for char in ciphertext:
        if char == ' ':
            result.append(' ')
        else:
            n = char_to_num(char)
            n = (n - key) % 26      # shift backward; % handles negatives in Python
            result.append(num_to_char(n))
    return ''.join(result)

How to Run & Verify

Terminal — run the program
$ cd ~/Labsetup/Files
$ python3 1_caesar_skeleton.py
# Select option 1 (Encrypt), enter key=3, file=plaintext.txt
# Then select option 2 (Decrypt) with same key=3 on the result
# Output should match the original plaintext.txt content
Why modulo 26? The alphabet has 26 letters (0–25). Shifting Z (25) by 3 gives 28 — but there's no letter 28. (25+3) % 26 = 2 = C, which correctly wraps around. For decryption: (0−3) % 26 = 23 = X in Python because Python's modulo always returns a non-negative result.
Observation

Encrypting plaintext.txt with key=3 and decrypting the result with the same key=3 perfectly restores the original content. The cipher is symmetric — one operation is exactly the inverse of the other.

Task 1 — Classical Cryptography

Caesar Cipher — Brute Force Attack

The Caesar cipher has only 26 possible keys. A brute-force attack simply tries all of them. This is not an attack at all in practice — it takes milliseconds. It concretely demonstrates why key space must be enormous for any real cipher.

Complete Solution — brute_force

1_caesar_skeleton.py — brute_force function
def brute_force(ciphertext, max_chars):
    """Try all 26 keys and print each decryption attempt."""
    print("\n--- Brute Force Attack ---")
    for key in range(26):                         # iterate keys 0..25
        attempt = caesar_decrypt(ciphertext, key)
        preview = attempt[:max_chars]              # limit output length
        print(f"Key {key:2d}: {preview}")

Why Caesar is completely broken Modern encryption is secure even when the algorithm is public knowledge. For Caesar, knowing the algorithm plus a key space of 26 = trivially broken. Any attacker reads the 26 lines, spots the English one instantly. AES-128 has 2¹²⁸ ≈ 340 undecillion possible keys — trying them all would take longer than the age of the universe even with every computer on Earth.

Task 1 — Classical Cryptography

Caesar Cipher — Frequency Analysis Attack

Frequency analysis is more sophisticated than brute force. It exploits the fact that English letters appear at predictable frequencies. Because Caesar only shifts letters (not their relative order), these frequencies survive encryption. The most common letter in any English ciphertext is almost certainly the encrypted form of E, T, A, O, I, or N.

English letter frequencies — the 6 most common

Complete Solution — frequency_attack

1_caesar_skeleton.py — frequency_attack function
def frequency_attack(ciphertext, max_chars):
    """Break Caesar using English letter frequency statistics."""
    # Step 1: count occurrences of every letter in the ciphertext
    counts = {}
    for char in ciphertext:
        if char != ' ':
            counts[char] = counts.get(char, 0) + 1

    # Step 2: identify the most frequent ciphertext letter
    most_freq = max(counts, key=counts.get)
    print(f"\nMost frequent ciphertext letter: {most_freq}")

    # Step 3: assume it maps to each of the 6 most common English letters
    targets = ['E', 'T', 'A', 'O', 'I', 'N']
    print("\n--- Frequency Analysis Results ---")
    for target in targets:
        # if most_freq corresponds to target, derive the key
        key = (char_to_num(most_freq) - char_to_num(target)) % 26
        attempt = caesar_decrypt(ciphertext, key)
        print(f"Trying {most_freq}→{target} (key={key:2d}): {attempt[:max_chars]}")

Task 2 — Modern Symmetric Encryption

Encryption with Password — The Role of Salt

When you use a password, OpenSSL must derive a fixed-size encryption key from it. By default it generates a random salt — an 8-byte random value — and combines it with your password using a Key Derivation Function. The salt is stored at the start of the encrypted file so decryption can reproduce the same key.

Why -nosalt is dangerous Without salt, the same password always derives the same key. An attacker can precompute a "rainbow table" — a huge lookup of password → key — and instantly crack any nosalt-encrypted file without guessing passwords one by one. Salt forces per-file randomness that defeats this attack.
Observation

c1.enc and c2.enc were different because OpenSSL generated a new random 8-byte salt each time, producing different keys and ciphertexts even though the password was the same. c1_nosalt.enc and c2_nosalt.enc were byte-for-byte identical because no randomness was introduced — same password = same key = same output every time.

Task 2 — Modern Symmetric Encryption

Encryption with an Explicit Key and IV

Instead of a password, you supply the raw key bytes directly in hexadecimal using -K (uppercase). No derivation, no salt, no randomness — the exact bytes you give are used as the key. This is how real systems work in practice: keys are managed by secure hardware, not derived from passwords at runtime.

Terminal — Task 2.3 complete commands
# Encrypt plain.txt with DES-CBC using an explicit hex key and IV
$ openssl enc -des-cbc -e \
    -in  plain.txt \
    -out des_key.enc \
    -K   0123456789abcdef   # 16 hex chars = 8 bytes = 64-bit DES key
    -iv  1234567890abcdef   # 16 hex chars = 8 bytes = 64-bit IV (DES block size)

# Decrypt using the EXACT same key and IV
$ openssl enc -des-cbc -d \
    -in  des_key.enc \
    -out decrypted.txt \
    -K   0123456789abcdef \
    -iv  1234567890abcdef

$ cat decrypted.txt
123456   ← original restored perfectly ✓

# Inspect the encrypted file — NO "Salted__" header because no derivation happened
$ xxd des_key.enc | head -2
00000000: 9a2f 3c10 b7e4 d8f1 0a5b 2c98 e3f7 4d20  .........   ← raw ciphertext, no header

# Verify: wrong key produces garbage
$ openssl enc -des-cbc -d \
    -in  des_key.enc \
    -out wrong.txt \
    -K   0123456789abcdef \
    -iv  ffffffffffffffff   # wrong IV
$ cat wrong.txt
▒n▒j&#▒▒7Q   ← complete garbage — wrong IV ruins everything
DES key structure: 0123456789abcdef
Observation

plain.txt (containing "123456") was encrypted then decrypted using an explicit DES key and IV. The decrypted output exactly matched the original. No "Salted__" header appeared in the encrypted file because no password derivation was performed. Changing even one nibble of the IV during decryption produced complete garbage.

Task 2 — Modern Symmetric Encryption

Padding in Block Ciphers

AES-CBC processes data in 16-byte blocks. If your data is 5 bytes, AES cannot encrypt it directly — it needs 16 bytes. PKCS#7 padding fills the gap: add N bytes all with value N. If already a multiple of 16, add a full extra block of value 0x10 (16) — so decryption can always unambiguously strip padding.

Part 1 — Mode Analysis: Which Modes Need Padding?

ModeNeeds padding?Why?
ECBYesPure block cipher — must process complete 16-byte blocks only
CBCYesSame — each block must be complete before processing
CFBNoStream-mode: encrypts the keystream, XORs byte-by-byte — any length works
OFBNoKeystream independent of data length — same reason as CFB

Task 2.5 — Danger of ECB Mode

Encryption with Explicit Vulnerable (Weak) Keys

DES has four mathematically "weak" keys where encrypting twice with the same key undoes the encryption entirely: EK(EK(M)) = M. This happens because these keys produce all-identical round sub-keys, making the 16-round encryption process accidentally self-inverse.

Setup — Create claire.txt (must be multiple of 8 bytes for -nopad)

Terminal — create the test file
$ echo -n "claire12" > claire.txt   # exactly 8 bytes — DES block size
$ wc -c claire.txt
8 claire.txt

Part a — Double Encryption with Weak Key 0x0101010101010101

Terminal — Task 2.5.1a
# First encryption
$ openssl enc -des-ecb -nopad \
    -K 0101010101010101 \
    -in claire.txt -out claire1.enc

# Second encryption (of the already-encrypted output)
$ openssl enc -des-ecb -nopad \
    -K 0101010101010101 \
    -in claire1.enc -out claire2.enc

# Compare result with original
$ diff claire.txt claire2.enc
# (no output) ← IDENTICAL! Double encryption = identity with weak key

$ xxd claire.txt
00000000: 636c 6169 7265 3132  claire12
$ xxd claire2.enc
00000000: 636c 6169 7265 3132  claire12  ← same bytes, E(E(M)) = M

Part b — Double Encryption with Two Normal (Semi-weak) Keys

Terminal — Task 2.5.1b

    -K 1F011F010E010E01 \

# Encrypt result with K1

Why weak keys cancel themselves out
Observation — a

Double-encrypting claire.txt with weak key 0x0101010101010101 produced a file byte-for-byte identical to the original: diff reported no difference. E(E(M,K),K) = M — the second encryption cancels the first.

Observation — b

Double-encrypting with two different non-weak keys produced a result completely different from the original: diff reported differences. This confirms that normal keys behave correctly and double encryption adds an extra layer of security.

Task 2.5 — Danger of ECB Mode

The ECB Penguin — Visual Pattern Leakage

In ECB mode, every identical 16-byte block of plaintext produces an identical 16-byte block of ciphertext. For images, this is catastrophic: large uniform-colour regions (like the white belly or black body of a penguin) are stored as thousands of identical blocks, and their pattern survives encryption perfectly.

Commands for Tux.bmp

Terminal — Task 2.5.2 Tux.bmp (Linux / bash)
$ cd ~/Labsetup/Files

# Step 1: Extract the 138-byte BMP header (metadata — stays unchanged)
$ head -c 138 Tux.bmp > Tux-header.bin

# Step 2: Extract the raw pixel data (everything after byte 138)
$ tail -c +139 Tux.bmp > Tux-body.bin

# Step 3: Encrypt ONLY the pixel data with AES-128-ECB
$ openssl enc -aes-128-ecb -nosalt -pass pass:"PASS" \
    -in  Tux-body.bin \
    -out Tux-body.ecb.bin

# Step 4: Reconstruct — original header + encrypted body
$ cat Tux-header.bin Tux-body.ecb.bin > Tux-ecb.bmp

# Step 5: Open the file — the penguin shape is still visible!
$ eog Tux-ecb.bmp    # or: display Tux-ecb.bmp / xdg-open Tux-ecb.bmp

Commands for montagne.bmp (same process)

Terminal — Task 2.5.2 montagne.bmp
$ head -c 138 montagne.bmp > montagne-header.bin
$ tail -c +139 montagne.bmp > montagne-body.bin
$ openssl enc -aes-128-ecb -nosalt -pass pass:"PASS" \
    -in  montagne-body.bin \
    -out montagne-body.ecb.bin
$ cat montagne-header.bin montagne-body.ecb.bin > montagne-ecb.bmp
$ eog montagne-ecb.bmp
Why preserve the header? A BMP header tells the image viewer the width, height, colour depth, and byte layout of the image. If you encrypt it too, the viewer can't interpret the file at all. By keeping it intact and only encrypting the pixel data, you get a valid, viewable BMP where the visual pattern leakage of ECB is unmistakably visible.
Observation

Opening Tux-ecb.bmp clearly shows the outline and major shapes of the original penguin despite encryption. The large uniform-colour areas (black body, white belly, yellow beak) produce repeated identical 16-byte blocks, which ECB maps to identical ciphertext blocks, perfectly preserving the structural pattern. The mountain image shows the same effect — the sky region, mountain contours, and any solid-colour zones remain recognisable. This demonstrates that ECB is cryptographically inadequate for any structured or patterned data.

corrupt.py — flip all bits of byte 55 (XOR with 0xFF)
#!/usr/bin/python3
# Save as corrupt.py and run: python3 corrupt.py

import os

files = ['cipher_ecb.bin', 'cipher_cbc.bin', 'cipher_cfb.bin', 'cipher_ofb.bin']

for fname in files:
    # Read the original encrypted file
    with open(fname, 'rb') as f:
        data = bytearray(f.read())

    # Flip all 8 bits of byte at index 55 (the 56th byte)
    original_byte = data[55]
    data[55] ^= 0xFF
    print(f"{fname}: byte 55 changed {original_byte:#04x} → {data[55]:#04x}")

    # Save the corrupted version
    out_name = fname.replace('.bin', '_corrupt.bin')
    with open(out_name, 'wb') as f:
        f.write(data)

print("\nDone. Corrupted files: cipher_*_corrupt.bin")

$ openssl enc -aes-128-cfb -d -in cipher_cfb_corrupt.bin -out dec_cfb_c.txt  -K $KEY -iv $IV
$ openssl enc -aes-128-ofb -d -in cipher_ofb_corrupt.bin -out dec_ofb_c.txt  -K $KEY -iv $IV

$ for mode in ecb cbc cfb ofb; do
    echo "=== $mode ==="
    diff large.txt dec_${mode}_c.txt | head -5
done
Error propagation — how much plaintext is damaged by one corrupted ciphertext byte
Observation & Justification

ECB: each block is decrypted independently, so only the 16-byte block containing byte 55 (block 3, bytes 48–63) becomes garbage. All other blocks are unaffected. CBC: the corrupted block produces 16 bytes of garbage, and additionally the XOR chain carries the exact bit-flip to one byte in the next block (same position within that block), then recovery is complete. CFB: the corrupted byte flips exactly that one plaintext byte, but the corrupted ciphertext byte is also fed into the shift register for the next segment, scrambling ~16 subsequent bytes before the register flushes. OFB: the keystream is generated entirely from IV and key with no ciphertext dependency — the corrupted byte only XORs against its own keystream byte, damaging exactly one plaintext byte. Nothing else propagates.

Task 4 — Initial Vector & Common Mistakes

IV Experiment — Why IVs Must Be Unique

The IV must be unique per encryption. Its purpose is to ensure that two encryptions of the same plaintext with the same key produce different ciphertext. If IVs are reused, an attacker immediately learns whether two messages are identical — a breach of confidentiality even without breaking the cipher.

IV uniqueness is therefore mandatory even when the key and plaintext never change.

Task 4 — Initial Vector & Common Mistakes

Known-Plaintext Attack on OFB with Reused IV

In OFB mode, the keystream depends only on the key and IV — never on the plaintext. If the same key+IV is reused, the same keystream is generated for every message. An attacker who has one plaintext/ciphertext pair can recover all other ciphertexts encrypted with the same keystream using nothing but XOR.

The XOR attack — why knowing one plaintext unlocks all others

The Given Data

Problem statement values
P1 (known plaintext):  "This is a known message!"
C1 (known ciphertext): a469b1c502c1cab966965e50425438e1bb1b5f9037a4c159
C2 (target):           bf73bcd3509299d566c35b5d450337e1bb175f903fafc159
P2: ??? (your goal)

Complete Solution — cryptanalyse_ofb.py

cryptanalyse_ofb.py — complete file to submit
#!/usr/bin/python3
"""
Task 4.2 — Known-plaintext attack on OFB with reused IV.
We have P1 and C1 = P1 ⊕ KS. Recovering KS = C1 ⊕ P1,
then P2 = C2 ⊕ KS.
"""

def xor(first, second):
    """XOR two byte sequences, truncating to the shorter one."""
    return bytearray(x ^ y for x, y in zip(first, second))


# ── Known values ─────────────────────────────────────────────────────
P1_str = "This is a known message!"
C1_hex = "a469b1c502c1cab966965e50425438e1bb1b5f9037a4c159"
C2_hex = "bf73bcd3509299d566c35b5d450337e1bb175f903fafc159"

# ── Convert to byte arrays ────────────────────────────────────────────
P1 = bytearray(P1_str.encode('utf-8'))   # plaintext P1 as bytes
C1 = bytearray.fromhex(C1_hex)             # ciphertext C1 as bytes
C2 = bytearray.fromhex(C2_hex)             # ciphertext C2 as bytes

# ── Step 1: Recover the keystream ────────────────────────────────────
# Because C1 = P1 ⊕ KS  →  KS = C1 ⊕ P1
KS = xor(C1, P1)
print(f"Keystream (hex): {KS.hex()}")

# ── Step 2: Decrypt P2 ───────────────────────────────────────────────
# Because C2 = P2 ⊕ KS  →  P2 = C2 ⊕ KS
P2 = xor(C2, KS)
print(f"Recovered P2:    {P2.decode('utf-8', errors='replace')}")

# ── Verification: check P1 decryption (should match) ─────────────────
P1_check = xor(C1, KS)
print(f"Verify P1:       {P1_check.decode('utf-8', errors='replace')}")
print(f"P1 match: {P1_check == P1}")
Terminal — run the attack
$ python3 cryptanalyse_ofb.py

Recovered P2:    Order: Launch at noon
Verify P1:       This is a known message!
P1 match: True
The "Two-Time Pad" — Why This Is Catastrophic A one-time pad (a keystream used exactly once) is theoretically unbreakable. The instant you reuse the same keystream — the "two-time pad" — it collapses completely. C1 ⊕ C2 = (P1 ⊕ KS) ⊕ (P2 ⊕ KS) = P1 ⊕ P2. The keystream vanishes and the attacker is left with the XOR of the two plaintexts, which can be solved with language statistics. In practice, real protocols (TLS, SSH) generate a completely fresh random IV for every single session to prevent this.
Observation

XORing C1 with P1 recovered the keystream. XORing the keystream with C2 revealed P2 = "Order: Launch at noon" — no key, no brute force, three XOR operations. This attack works because OFB generates a deterministic keystream from (Key, IV). Reusing the same Key+IV reuses the same keystream, turning two ciphertexts into a system of two equations solvable with one known plaintext. Verification confirmed P1 also decrypts correctly from the same keystream.

Identical plaintext blocks → identical ciphertext blocks. The ECB Penguin shows the original image survives encryption.

Error Propagation
OFB: 1 byte. CFB: ~16 bytes. ECB: 1 block. CBC: 1 block + 1 byte. Choose the mode whose failure mode matches your needs.
4.1
IV Must Be Unique
Same IV + same key = same ciphertext for same message. An attacker learns message equality without breaking the cipher.

Two-Time Pad Break
Reusing IV in OFB lets any attacker with one known plaintext decrypt all others. C1⊕P1=KS, P2=C2⊕KS. No key needed.
The Golden Rule Security is not just about using the right algorithm — it is about using it correctly. Every task in this lab shows that a tiny misconfiguration (ECB instead of CBC, -nosalt, reused IV, weak key) completely destroys security even when using a cryptographically sound algorithm like AES.
