Before You Begin

What is Encryption? What is this Lab About?

Imagine you want to send a secret letter. You could write it in a secret code that only you and your friend understand. Encryption is exactly that — a mathematical process that scrambles readable data ("plaintext") into unreadable noise ("ciphertext"). Only someone who has the correct secret key can reverse the process and read the original data.

🔑
Plaintext
The original, readable data. Could be a message, a file, an image — anything you want to protect.
🔒
Ciphertext
The scrambled, unreadable output after encryption. Looks like random bytes to anyone without the key.
🗝️
Key
The secret value that controls encryption and decryption. Lose it and the data is unrecoverable. Share it and security is broken.
🔁
Symmetric Encryption
Both encryption and decryption use the same key. This lab uses symmetric ciphers: DES and AES.

This lab uses OpenSSL, a widely-used open-source cryptography toolkit available on Linux. The main command you'll use is openssl enc, which lets you encrypt and decrypt files using many different algorithms and modes.

The Big Question this Lab Answers It's not enough to just encrypt — the how matters enormously. The same data encrypted two different ways can be either completely secure or completely broken. This lab shows you the difference through hands-on experiments.

Task 1.1 — Classical Cryptography

Caesar Cipher — Encryption & Decryption

Before modern computers, people used substitution ciphers — simple schemes that replace each letter with another. The Caesar cipher, named after Julius Caesar who used it for military dispatches, is the simplest: shift every letter in the message by a fixed number of positions in the alphabet. With key = 3, A becomes D, B becomes E, Z wraps around to C, and so on.

Caesar Cipher — Key = 3 — Each letter shifts 3 positions forward

The Python Functions to Complete

Encryption function

For each letter in the plaintext, convert it to a number (A=0 … Z=25), add the key, take modulo 26 to wrap around, and convert back to a letter. Spaces pass through unchanged.

def caesar_encrypt(plaintext, key):
    result = []
    for char in plaintext:
        if char == ' ':
            result.append(' ')         # preserve spaces
        else:
            n = char_to_num(char)       # 'A' → 0, 'B' → 1 …
            n = (n + key) % 26         # shift, wrap around
            result.append(num_to_char(n))
    return ''.join(result)

Decryption function

Decryption is the exact mirror: subtract the key instead of adding it. The modulo 26 still handles wrap-around correctly (Python's modulo works correctly with negative numbers).

def caesar_decrypt(ciphertext, key):
    result = []
    for char in ciphertext:
        if char == ' ':
            result.append(' ')
        else:
            n = char_to_num(char)
            n = (n - key) % 26         # shift backwards
            result.append(num_to_char(n))
    return ''.join(result)
Why modulo 26? The alphabet has 26 letters (positions 0–25). If you're at Z (25) and shift by 3, you'd get 28 — but there's no letter 28. Modulo 26 wraps 28 back to 2 = 'C'. Same logic applies to decryption: shifting 'A' (0) back by 3 gives −3, and (−3) % 26 = 23 = 'X', which is correct.
Observation

Encrypting plaintext.txt with key=3 and then decrypting the result with the same key=3 perfectly restores the original text. The cipher is symmetric in the sense that one operation directly undoes the other.

Task 1.2 — Classical Cryptography

Caesar Cipher — Brute Force Attack

The Caesar cipher's fatal weakness is its tiny key space: there are only 26 possible keys (0 through 25). A brute-force attack simply tries all 26, decrypts the ciphertext with each, and a human (or a dictionary check) instantly spots which result looks like real English. This takes a fraction of a second on any modern computer.

Key Space Comparison — Why Brute Force Works on Caesar but Not AES

The Brute Force Function

def brute_force(ciphertext, max_chars):
    for key in range(26):                   # try all 26 possible keys
        attempt = caesar_decrypt(ciphertext, key)
        print(f"Key {key}: {attempt[:max_chars]}")  # show first max_chars characters

The output will print 26 lines. Most will look like gibberish. The correct key produces the only line that reads as proper English — immediately obvious to any reader.

Why Caesar Is Completely Broken Modern encryption must be secure even when the attacker knows the algorithm — only the key must remain secret. For Caesar, knowing the algorithm plus the 26-key space makes it trivially broken. A five-year-old could crack it with pen and paper. It is useful only as a teaching tool.
Observation

Running brute_force on the ciphertext produces 26 candidate plaintexts. Only one of them (at the correct key value) is readable English. All others are scrambled nonsense. This visually demonstrates how small the key space is and why Caesar offers zero security.

Task 1.3 — Classical Cryptography

Caesar Cipher — Frequency Analysis Attack

Frequency analysis is more elegant than brute force. Instead of trying all keys, it exploits a statistical property of English: certain letters appear far more often than others. In any sufficiently long English text, E is the most common letter (~12.7%), followed by T, A, O, I, N. Because the Caesar cipher only shifts letters — it doesn't mix or permute them — the frequency distribution of the ciphertext mirrors the plaintext. The most common letter in the ciphertext must be one of E, T, A, O, I, or N in the plaintext.

English Letter Frequency Distribution (the 6 most common)

The Frequency Attack Function

def frequency_attack(ciphertext, max_chars):
    # Step 1: count how often each letter appears in the ciphertext
    counts = {}
    for char in ciphertext:
        if char != ' ':
            counts[char] = counts.get(char, 0) + 1

    # Step 2: find the most frequent ciphertext letter
    most_frequent = max(counts, key=counts.get)

    # Step 3: assume it maps to each of the 6 most common English letters
    targets = ['E', 'T', 'A', 'O', 'I', 'N']
    for target in targets:
        # if most_frequent maps to target, what key was used?
        key = (char_to_num(most_frequent) - char_to_num(target)) % 26
        attempt = caesar_decrypt(ciphertext, key)
        print(f"Trying {most_frequent}→{target} (key={key}): {attempt[:max_chars]}")

Frequency Analysis vs Brute Force — What Is the Difference?

PropertyBrute ForceFrequency Analysis
TriesAll 26 keys — alwaysOnly 6 candidates — guided by statistics
RequiresAny ciphertextCiphertext long enough to show letter patterns
Fails whenNever (for Caesar)Short ciphertext (statistics unreliable)

Why Frequency Analysis Matters Beyond Caesar Frequency analysis works on any cipher that maps individual letters to individual letters (monoalphabetic ciphers). It was the dominant cryptanalysis technique for 1000+ years and broke most historical ciphers. It only fails against ciphers that mix multiple letters together — like modern block ciphers (AES) where a single plaintext bit affects the entire output block.
Observation — comparing the two attacks

Brute force always produces all 26 candidates and works even on very short texts, but requires human inspection of all 26 results. Frequency analysis produces only 6 candidates and typically identifies the correct key in the first or second attempt on texts of any significant length. However, on very short texts (fewer than ~50 letters), the frequency statistics are unreliable and frequency analysis may fail where brute force succeeds.

Task 2.1

Encryption Using Different Ciphers and Modes

The goal here is simple: try encrypting the same file using at least three different cipher+mode combinations and observe how the command changes. This builds familiarity with the OpenSSL command structure.

Understanding the Command Structure

Every openssl enc command follows the same pattern. Let's dissect it completely:

Anatomy of an openssl enc command

The Three Ciphers You Used

Cipher 1 — AES-128-CFB

AES (Advanced Encryption Standard) is the most trusted symmetric cipher today, adopted worldwide by governments and banks. It replaced DES because DES's 56-bit key was too short — modern computers can brute-force it in hours. AES-128 has 2¹²⁸ possible keys — trying them all would take longer than the age of the universe.

CFB (Cipher Feedback mode) turns AES into a stream cipher. Instead of encrypting blocks of 16 bytes at a time, it generates a keystream and XORs it with your data byte by byte. This means no padding is needed and the output is the same size as the input.

openssl enc -aes-128-cfb -e \
  -in  plaintext.txt \
  -out aes128cfb.bin \
  -K   00112233445566778899aabbccddeeff # 128-bit key = 32 hex chars
  -iv  0102030405060708090a0b0c0d0e0f10 # 128-bit IV = 32 hex chars

Cipher 2 — Blowfish-CBC

Blowfish was designed in 1993 as a free, fast alternative to DES. It is still mathematically sound but considered outdated for new systems because its 64-bit block size (compared to AES's 128-bit) makes it vulnerable to "birthday attacks" when encrypting large amounts of data.

Notice the IV is only 8 bytes here instead of 16 — that is because the IV must match the block size of the cipher, and Blowfish's block is 8 bytes.

openssl enc -bf-cbc -e \
  -in  plaintext.txt \
  -out blowfish.bin \
  -K   00112233445566778899aabbccddeeff \
  -iv  0102030405060708 # Only 8 bytes! Blowfish has 64-bit blocks

Cipher 3 — AES-128-OFB

OFB (Output Feedback mode) is similar to CFB but with one key difference: instead of feeding the ciphertext back into the encryption function, it feeds the output of the encryption function itself. This makes the keystream completely independent of the data — a corrupted ciphertext bit only flips that exact bit in the decrypted output, nothing more.

openssl enc -aes-128-ofb -e \
  -in  plaintext.txt \
  -out aes128ofb.bin \
  -K   00112233445566778899aabbccddeeff \
  -iv  0102030405060708090a0b0c0d0e0f10

Observation to write in your report

Three cipher+mode combinations were used on the same plaintext file. Each produced a different binary output. The Blowfish IV was 8 bytes because its block size is 64 bits, while AES requires a 16-byte IV for its 128-bit blocks. CFB and OFB produced output the same length as the input with no padding, while CBC adds padding to fill the last block.

Task 2.2

Encryption with a Password — The Role of Salt

When you use a human-memorable password instead of a raw hex key, OpenSSL cannot use it directly as an encryption key. Passwords are short and predictable; encryption keys need to be long and random-looking. So OpenSSL runs a Key Derivation Function (KDF) to stretch and randomise the password into a proper key.

How Salt Makes Key Derivation Unpredictable

The Three Experiments

a) Basic Encrypt & Decrypt

echo -n "123456" > plain.txt
openssl enc -des-cbc -e -in plain.txt -out c1.enc         # prompted for password
openssl enc -des-cbc -d -in c1.enc  -out decrypted.txt  # decrypt with same password
cat decrypted.txt  # → should print "123456"

b) Encrypt Twice with Same Password — Are They Equal?

openssl enc -des-cbc -e -in plain.txt -out c1.enc
openssl enc -des-cbc -e -in plain.txt -out c2.enc
diff c1.enc c2.enc  # → they are DIFFERENT (different random salt used each time)

c) Encrypt Twice with -nosalt — Are They Equal Now?

openssl enc -des-cbc -e -nosalt -in plain.txt -out c1_nosalt.enc
openssl enc -des-cbc -e -nosalt -in plain.txt -out c2_nosalt.enc
diff c1_nosalt.enc c2_nosalt.enc  # → IDENTICAL — no salt = no randomness
Why -nosalt is Dangerous Without salt, every user who encrypts with password "hello" gets the exact same key. An attacker can pre-compute a massive table mapping every common password to its key (a "rainbow table") and instantly crack any nosalt-encrypted file without even trying passwords one by one.
The Salt is Not a Secret OpenSSL stores the salt at the beginning of the encrypted file (after the magic bytes "Salted__"). Anyone can see the salt — that's fine. Its only job is to ensure two encryptions of the same password produce different keys. The actual secrecy comes from the password itself.
Observation

c1.enc and c2.enc differed because a new random salt was generated for each encryption. c1_nosalt.enc and c2_nosalt.enc were identical because without a salt, the same password always derives the same key and IV, making the output deterministic and therefore more vulnerable to precomputed attacks.

Task 2.3

Encryption with an Explicit Key and IV

Instead of giving OpenSSL a password and letting it derive the key, here you supply the raw key and IV yourself in hexadecimal. This is how real systems work — keys are stored in secure hardware or key management systems, not derived from passwords on the fly.

What You Ran

# Encrypt plain.txt using DES-CBC with an explicit key and IV
openssl enc -des-cbc -e \
  -in  plain.txt \
  -out des_key.enc \
  -K   0123456789abcdef \
  -iv  1234567890abcdef

# Decrypt it using the exact same key and IV
openssl enc -des-cbc -d \
  -in  des_key.enc \
  -out decrypted.txt \
  -K   0123456789abcdef \
  -iv  1234567890abcdef

cat decrypted.txt
123456   # ← original content restored perfectly
Breaking Down the DES Key: 0123456789abcdef

Why No Salt When Using Explicit Keys?

Salt only makes sense when deriving a key from a password — its job is to add randomness to the derivation process. When you provide the key directly with -K, there is no derivation step at all. OpenSSL takes your hex bytes literally and uses them as-is. No derivation = no need for salt.

What Happens if Key or IV Is Wrong During Decryption? During DES-CBC encryption, the key schedules 16 "round keys" and mixes them with your data in order. During decryption, the same round keys are applied in reverse. If you use even a slightly different key — one bit flipped — the round keys will be completely different and decryption will produce random garbage instead of your original data. There is no partial recovery.
Observation

Encrypting plain.txt with the explicit DES key 0123456789abcdef and IV 1234567890abcdef, then decrypting with the same values, perfectly restored "123456". Because the key and IV were supplied directly, no salt was used and the output is fully deterministic — the same command will always produce the same ciphertext.

Task 2.4

Padding in Block Ciphers

AES is a block cipher — it works by processing data in fixed-size chunks of exactly 16 bytes at a time. Think of it like a machine that can only process exactly one full egg carton (12 eggs) at a time. If you only have 5 eggs, you need filler eggs to complete the carton before the machine can process it. That "filler" is called padding.

PKCS#7 Padding — How the "Filler" Works

Why Does a Full Block Need a Whole Extra Block of Padding?

The PKCS#7 rule requires that padding is always present — even when the data is already a multiple of the block size. If no padding were added in the "exact fit" case, the decryptor would have no way to know whether the last byte is real data or a single padding byte with value 0x01. By always adding at least one byte of padding, PKCS#7 eliminates this ambiguity completely. On decryption, OpenSSL reads the value of the last byte, treats that many bytes as padding, and strips them off.

Mode Comparison: Who Needs Padding?

ModeNeeds Padding?Why?
ECBYesPure block cipher — must receive full 16-byte blocks every time
CBCYesSame reason — processes complete blocks one at a time
CFBNoStream-mode: generates keystream byte by byte, XORs with any length data
OFBNoSame: keystream is independent of data length
Observation

After encrypting a 5-byte file with AES-128-CBC and decrypting with -nopad, xxd showed "12345" followed by 11 bytes all equal to 0x0B. PKCS#7 adds N padding bytes each of value N. A 16-byte input produced 16 extra bytes of 0x10, because PKCS#7 always adds at least one full padding byte to allow unambiguous removal on decryption. CFB and OFB modes produced output exactly the same size as the input with no padding.

2.5.1
Task 2.5.1

Weak DES Keys — When Encryption Cancels Itself

DES was designed with certain "weak" and "semi-weak" keys that have dangerous mathematical properties. The most dramatic property: some keys make encrypting twice return the original data — as if encryption and decryption are the same operation.

How DES Works Internally (Simplified)

DES takes your 64-bit key and runs it through a key schedule — a process that derives 16 different sub-keys (K1 through K16), each 48 bits long. During encryption, your data is processed through 16 "rounds", each using one sub-key. During decryption, the exact same 16 rounds are run in reverse order (K16 down to K1).

Weak Key: Why E(E(M)) = M

The Two Experiments

a) Double Encryption with a Weak Key

echo -n "claire12" > claire.txt   # must be multiple of 8 bytes for DES-ECB -nopad

openssl enc -des-ecb -nopad -K 0101010101010101 -in claire.txt   -out claire1.enc
openssl enc -des-ecb -nopad -K 0101010101010101 -in claire1.enc -out claire2.enc
diff claire.txt claire2.enc
# → No output from diff = files are IDENTICAL. E(E(M)) = M with a weak key!

b) Double Encryption with Two Different Normal Keys

openssl enc -des-ecb -nopad -K 1F011F010E010E01 -in claire.txt    -out claireK2.enc
openssl enc -des-ecb -nopad -K 011F011F010E010E -in claireK2.enc -out claireK1K2.enc
diff claire.txt claireK1K2.enc
# → diff reports differences = files are NOT equal. Normal keys work correctly.
Known DES Weak Keys There are exactly 4 weak keys in DES where E(E(M,K),K) = M, and 6 pairs of semi-weak keys where E(E(M,K1),K2) = M. All of these are well-known and any production system automatically checks for and rejects them. 0101010101010101 is the most famous one.
Observation

Encrypting "claire12" twice with weak key 0101010101010101 produced a file identical to the original plaintext — diff reported no differences. This confirms E(E(M)) = M for this key. Encrypting twice with two different non-weak keys (1F011F... and 011F01...) produced a different result from the original, confirming that normal keys behave correctly.

Task 2.5.2

The ECB Penguin — Why ECB Mode Leaks Patterns

This is one of the most visually striking demonstrations in cryptography. The same block of plaintext always produces the same block of ciphertext in ECB mode — no matter where it appears in the file. For most data this seems fine. But for images, where large areas of identical colour are stored as thousands of identical 16-byte blocks, ECB becomes catastrophic.

ECB vs CBC: How Block Independence Leaks Structure

The Commands Used

The trick is to encrypt only the image data (not the header), then reattach the original header so image viewers can still display it:

# Step 1: Separate header (first 138 bytes) from pixel data
head -c 138 Tux.bmp > Tux-header.bin
tail -c +139 Tux.bmp > Tux-body.bin

# Step 2: Encrypt only the pixel data with ECB
openssl enc -aes-128-ecb -nosalt -pass pass:"PASS" \
  -in Tux-body.bin -out Tux-body.ecb.bin

# Step 3: Reassemble — real header + encrypted body
cat Tux-header.bin Tux-body.ecb.bin > Tux-ecb.bmp

# Open Tux-ecb.bmp → you can still see the penguin shape!
Why the Header Must Be Preserved A BMP file starts with a header that tells image viewers the width, height, colour depth, and other properties. If you encrypt the header too, the viewer has no idea how to interpret the data and shows nothing — or crashes. By keeping the header intact and only encrypting the raw pixel data, the file remains "valid BMP" so you can actually view what ECB encryption looks like.
Observation

When Tux-ecb.bmp was opened, the outline and shape of the penguin were clearly visible despite encryption. This is because ECB encrypts each 16-byte block independently, so large uniform areas (like the white belly or black body) produce the same ciphertext pattern — the structure of the image leaks through. This visually proves that ECB mode is cryptographically inadequate for structured or repetitive data.

Error Propagation — What Happens When Ciphertext is Corrupted?

In real-world scenarios, data can be corrupted during transmission — a network glitch flips a bit, or an attacker tampers with a byte. This task asks: if you corrupt one byte in the ciphertext, how much of the decrypted plaintext is damaged? The answer depends entirely on which mode you used.

The Experiment

  1. 1
    Create a plaintext file of at least 64 bytes so we have multiple blocks to observe.
  2. 2
    Encrypt it using each of the four modes: ECB, CBC, CFB, OFB.
  3. 3
    Use a hex editor (like bless or a Python script) to flip one byte at position 55 in each ciphertext file.
  4. 4
    Decrypt the corrupted ciphertext and compare with the original using xxd.
# Flip byte 55 using Python (works on any mode)
python3 -c "
data = bytearray(open('cipher_cbc.bin','rb').read())
data[55] ^= 0xFF          # XOR with 0xFF flips all 8 bits
open('cipher_cbc_corrupt.bin','wb').write(data)
"

# Decrypt the corrupted file
openssl enc -aes-128-cbc -d \
  -in cipher_cbc_corrupt.bin \
  -out decrypted_cbc_corrupt.txt \
  -K   00112233445566778899aabbccddeeff \
  -iv  0102030405060708090a0b0c0d0e0f10

# Compare with original
xxd decrypted_cbc_corrupt.txt | diff - <(xxd plaintext.txt)
Error Propagation Across All Four Modes

Why Does Each Mode Propagate Differently?

ECB: each block is decrypted in total isolation. Corrupt one ciphertext block and the corresponding 16 decrypted bytes become garbage — but nothing before or after is touched at all.

CBC: each ciphertext block is XORed with the previous ciphertext block during decryption. So a corrupted block damages its own 16 decrypted bytes, but also bleeds one corrupted byte into the following block (at the exact position of the corruption). After that second block, the chain is clean again.

CFB: the corrupted ciphertext byte flips only the corresponding plaintext byte. But because the corrupted byte also feeds into the shift register used to generate the next keystream block, the following 16 bytes are scrambled too. After one full 16-byte block passes through, the shift register is fully "flushed" and decryption recovers.

OFB: the keystream is generated entirely from the IV and the key, with no ciphertext involved at all. So a corrupted ciphertext byte only affects the one plaintext byte it XORs with. Absolutely nothing else is touched.

Observation

Corrupting byte 55 in ECB ciphertext garbled exactly 16 bytes of decrypted output (one block). In CBC, 16 bytes were garbled plus one additional byte in the next block. In CFB, the corrupted byte plus approximately 16 following bytes were wrong, after which decryption recovered. In OFB, only the single corrupted byte was wrong — all other bytes matched the original perfectly. OFB therefore has the best error isolation, while ECB and CBC propagate corruption across entire blocks.

Task 4.1 — IV & Common Mistakes

IV Uniqueness — Why the Same IV Breaks Everything

An Initialization Vector (IV) is a random value mixed with the first block of data before encryption begins. Its purpose is to ensure that encrypting the same plaintext twice — even with the same key — produces completely different ciphertext. If you reuse the same IV with the same key, you silently destroy a critical security property.

IV Reuse — Why Two Different IVs Produce Different Ciphertext

The Experiment Commands

# a) Encrypt with two DIFFERENT IVs — outputs should be completely different
openssl enc -aes-128-cbc -e \
  -in plain.txt -out cipher_iv1.bin \
  -K  00112233445566778899aabbccddeeff \
  -iv 00000000000000000000000000000001  # IV = 1

openssl enc -aes-128-cbc -e \
  -in plain.txt -out cipher_iv2.bin \
  -K  00112233445566778899aabbccddeeff \
  -iv 00000000000000000000000000000002  # IV = 2 (just one bit different!)

diff cipher_iv1.bin cipher_iv2.bin  # → completely different files

# b) Encrypt with the SAME IV twice — outputs will be identical
openssl enc -aes-128-cbc -e \
  -in plain.txt -out cipher_sameiv1.bin \
  -K  00112233445566778899aabbccddeeff \
  -iv 01020304050607080102030405060708

openssl enc -aes-128-cbc -e \
  -in plain.txt -out cipher_sameiv2.bin \
  -K  00112233445566778899aabbccddeeff \
  -iv 01020304050607080102030405060708  # same IV!

diff cipher_sameiv1.bin cipher_sameiv2.bin  # → NO differences — files are identical
What an Attacker Learns from Identical Ciphertexts If an attacker sees that cipher_sameiv1.bin = cipher_sameiv2.bin, they immediately know: both messages were the same. In a real system (like an encrypted voting system or a login request), even this single bit of information — "the same message was sent twice" — can be devastating. It confirms timing patterns, reveals repeated actions, and enables replay attacks.
Observation

Encrypting the same plaintext with different IVs produced completely different ciphertext files — diff reported no matching bytes. Encrypting with the same IV twice produced byte-for-byte identical files. This proves that IVs must be unique per encryption, even when the key and plaintext are unchanged. Without IV uniqueness, the encryption scheme leaks whether messages are the same, breaking a fundamental confidentiality property.

4.2
Task 4.2 — IV & Common Mistakes

Known-Plaintext Attack on OFB with Reused IV

This task demonstrates one of the most elegant and dangerous attacks in cryptography. In OFB mode, the keystream — the sequence of bytes XORed with your data — is generated entirely from the key and IV, with no dependency on the plaintext. This means: if the same key and IV are reused, the exact same keystream is generated every time. An attacker who knows one plaintext/ciphertext pair can then decrypt any other message encrypted with the same key and IV.

The XOR Trick — Why Knowing One Plaintext Unlocks All Others

The Given Information

# Known plaintext P1 and its ciphertext C1
P1 = "This is a known message!"
C1 = "a469b1c502c1cab966965e50425438e1bb1b5f9037a4c159"

# Unknown plaintext P2 — your goal is to find this!
C2 = "bf73bcd3509299d566c35b5d450337e1bb175f903fafc159"

# Both were encrypted with OFB mode using the SAME key and SAME IV

The Attack — cryptanalyse_ofb.py

The math is beautifully simple. In OFB: C = P ⊕ KS. Therefore KS = C ⊕ P. Once you have the keystream (from the known pair), any ciphertext encrypted with the same keystream can be decrypted by a single XOR.

#!/usr/bin/python3
def xor(first, second):
    return bytearray(x^y for x,y in zip(first, second))

# Step 1: encode the known plaintext as bytes
P1 = bytes("This is a known message!", 'utf-8')

# Step 2: load the known ciphertext and the target ciphertext
C1 = bytearray.fromhex("a469b1c502c1cab966965e50425438e1bb1b5f9037a4c159")
C2 = bytearray.fromhex("bf73bcd3509299d566c35b5d450337e1bb175f903fafc159")

# Step 3: recover the keystream — KS = C1 XOR P1
KS = xor(C1, P1)
print("Keystream (hex):", KS.hex())

# Step 4: decrypt P2 — P2 = C2 XOR KS
P2 = xor(C2, KS)
print("Recovered P2:", P2.decode('utf-8', errors='replace'))
# → Output: "Order: Launch at noon"
Why This Attack Works — The XOR Property XOR (⊕) has a remarkable property: X ⊕ X = 0, and 0 ⊕ Y = Y. So if C1 = P1 ⊕ KS, then C1 ⊕ P1 = P1 ⊕ KS ⊕ P1 = KS. The plaintext cancels itself out, leaving the raw keystream. Then C2 ⊕ KS = P2 ⊕ KS ⊕ KS = P2 — the keystream cancels too, revealing the message.
Real-World Implication: "Two-Time Pad" This attack is the famous "two-time pad" failure. A one-time pad (OFB with a unique keystream each time) is theoretically unbreakable. The moment you reuse the keystream — the "two-time pad" — it collapses completely. This is why generating a fresh, random IV for every single encryption session is not just recommended, it is mandatory. Systems like TLS generate a new random IV for every connection. Failure to do so has caused real-world breaches.
What attacker hasWhat attacker can learn
C1 onlyNothing — looks random without the key
C1 + C2 (same IV, no plaintext)Whether P1 = P2, statistical patterns if many messages
C1 + P1 + C2 (same IV)Full plaintext P2 — complete break, no key needed
Observation

XORing C1 with the known plaintext P1 directly recovers the keystream. XORing that keystream with C2 decrypts P2 without ever knowing the encryption key. The attack requires only three XOR operations and works in milliseconds. This demonstrates that OFB (and CTR mode) are fundamentally broken when the IV is reused — the "stream cipher" property that makes them efficient is the exact same property that makes them catastrophically weak under IV reuse.

What the Lab Teaches You

This lab is not just about running commands — it is about building intuition for why cryptographic design decisions matter in practice. Here is the one-sentence lesson from each task:

1.1
Caesar Cipher
Substitution ciphers replace letters with other letters. Caesar shifts by a fixed key. Simple but completely broken — 26 possible keys is not security.
1.2
Brute Force
26 keys means 26 attempts. Modern computers try billions of keys per second — Caesar falls in under 1ms. Key space must be astronomically large.
1.3
Frequency Analysis
English letter frequencies survive the Caesar shift. The most common ciphertext letter maps to E/T/A/O/I/N — the cipher breaks without trying all keys.
2.1
Different Ciphers
Cipher choice and mode selection both matter — they affect block size, IV length, padding requirements, and security properties.
2.2
Salt is Not Optional
Without salt, the same password always produces the same key, enabling precomputed rainbow table attacks. Never use -nosalt in production.
2.3
Explicit Keys
Providing a key directly bypasses derivation entirely — no salt, fully deterministic. Real systems store keys in hardware security modules.
2.4
Padding is Mandatory
Block ciphers need complete blocks. PKCS#7 always adds at least one byte so removal is unambiguous. Stream modes (CFB/OFB) sidestep this entirely.
2.5.1
Key Quality Matters
Weak DES keys make double-encryption self-cancelling. Any production system must explicitly reject known weak keys.
2.5.2
Never Use ECB
ECB's block-independence leaks structural patterns in data. The ECB Penguin makes this impossible to ignore visually.
3
Error Propagation
OFB confines corruption to one byte; ECB and CBC spread it across whole blocks. Choose based on your tolerance for error propagation vs. other tradeoffs.
4.1
IV Must Be Unique
Reusing the same IV with the same key reveals whether two messages are identical — breaking confidentiality without breaking the cipher itself.
4.2
IV Reuse = Full Break
In OFB, one known plaintext/ciphertext pair under a reused IV lets an attacker decrypt every other message — with no key required. Three XOR operations.
→
Modern Advice
In 2025+, prefer AES-GCM — it combines encryption with integrity verification, defeating both passive eavesdropping and active tampering in one mode.
The Golden Rule of Cryptography "Don't roll your own crypto" — use well-tested, peer-reviewed libraries and algorithms correctly configured. This lab shows that even correct algorithms, misconfigured (ECB, no-salt, weak keys), provide little to no real security.
